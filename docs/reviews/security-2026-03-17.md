# Security Review: WikiChain

**Date:** 2026-03-17
**Reviewer:** Claude Opus 4.6 (automated)
**Scope:** Full codebase
**Overall Rating:** LOW RISK

---

## Executive Summary

WikiChain is a client-side-only PWA with no backend, no API routes, no authentication, and no user-submitted content that renders as HTML. The attack surface is minimal. No critical or high-severity issues were found. A handful of low and informational findings are documented below.

---

## Findings

### LOW-1: localStorage Data Tampering

**Severity:** Low
**Location:** `components/WikiChainGame.jsx` — `loadState()` / `saveState()`

**Description:** All game state (XP, streak, history, onboarding) is stored in `localStorage` with no integrity checks. A user can open DevTools and modify their XP, streak, or mark puzzles as completed with perfect scores.

**Impact:** Cheating in a single-player casual game. No competitive leaderboard exists, so impact is negligible.

**Recommendation:** Acceptable for current scope. If leaderboards are ever added, move scoring to a server or add a signed hash to detect tampering.

---

### LOW-2: Deprecated `document.execCommand("copy")` Fallback

**Severity:** Low
**Location:** `components/WikiChainGame.jsx`, lines 306-313 (`handleShare`)

**Description:** The clipboard fallback path uses the deprecated `document.execCommand("copy")` API. While not a security vulnerability per se, the fallback creates a temporary `<textarea>` element and appends it to the DOM, which is a minor DOM manipulation pattern that could theoretically be intercepted in a compromised environment.

**Impact:** Minimal. The data being copied is non-sensitive (share text). The primary `navigator.clipboard.writeText` path is used first.

**Recommendation:** No action required. The fallback is reasonable for older browser support.

---

### LOW-3: Service Worker Caches All Routes

**Severity:** Low
**Location:** `public/sw.js` (auto-generated by `@ducanh2912/next-pwa`)

**Description:** The service worker uses `self.skipWaiting()` and pre-caches all static assets. The `skipWaiting` call means new service worker versions activate immediately without waiting for existing tabs to close, which could cause version mismatches between cached assets.

**Impact:** Could cause a brief inconsistency if a user has the app open when a deploy happens. Not a security issue, but a reliability note.

**Recommendation:** This is default PWA behavior and acceptable. If stale cache issues appear, consider adding a version check or cache-busting strategy.

---

### INFO-1: No Content Security Policy (CSP) Headers

**Severity:** Informational
**Location:** `next.config.mjs` / `app/layout.js`

**Description:** No Content Security Policy headers are configured. While the app has no user-generated content and no inline scripts (Next.js handles this), adding a CSP would provide defense-in-depth against XSS if the app evolves.

**Recommendation:** Consider adding CSP headers via `next.config.mjs`:
```js
const nextConfig = {
  reactStrictMode: true,
  headers: async () => [
    {
      source: "/(.*)",
      headers: [
        {
          key: "Content-Security-Policy",
          value: "default-src 'self'; script-src 'self' 'unsafe-inline' 'unsafe-eval'; style-src 'self' 'unsafe-inline'; font-src 'self' https://fonts.gstatic.com; img-src 'self' data:;"
        }
      ]
    }
  ]
};
```

---

### INFO-2: No X-Frame-Options or Frame-Ancestors Header

**Severity:** Informational
**Location:** `next.config.mjs`

**Description:** The app does not set `X-Frame-Options` or `frame-ancestors` CSP directive, meaning it could be embedded in an iframe on another site. For a game with no auth, this is not exploitable, but it is a best-practice gap.

**Recommendation:** Add `X-Frame-Options: DENY` header if iframe embedding is not desired.

---

### INFO-3: No Rate Limiting on Hint System (Client-Side Only)

**Severity:** Informational
**Location:** `components/WikiChainGame.jsx`, `handleHint()`

**Description:** The hint system deducts XP client-side. Since XP is stored in localStorage, users can give themselves unlimited XP and hints. This is inherent to the client-only architecture.

**Impact:** None for a casual single-player game.

**Recommendation:** No action needed unless competitive features are added.

---

## Positive Observations

1. **No backend attack surface.** The app is entirely static/client-side with no API routes, database connections, or server-side state.

2. **No user input rendered as HTML.** All content is hardcoded in `data/puzzles.js`. React's JSX escaping prevents XSS even if data were dynamic.

3. **localStorage access is properly guarded.** All `localStorage` calls check `typeof window !== "undefined"` and wrap `JSON.parse` in try/catch, preventing SSR crashes and malformed data errors.

4. **No secrets or credentials in the codebase.** No `.env` files, API keys, or tokens are present.

5. **`.gitignore` is well-configured.** Covers `.env*`, `.vercel`, `node_modules`, and generated PWA files.

6. **Minimal dependency footprint.** Only 4 production dependencies (`next`, `react`, `react-dom`, `@ducanh2912/next-pwa`), reducing supply chain risk.

7. **React strict mode enabled.** Helps catch potential issues during development.
